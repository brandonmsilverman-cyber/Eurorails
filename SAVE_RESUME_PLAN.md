# Save & Resume Implementation Plan

## Overview

Players can save an in-progress game, close their browsers, and return later to resume where they left off. Games survive server restarts. The feature is additive — it does not change how existing create/join/play flows work.

## Key Design Decisions

### Persistence: JSON files on disk

Save game state as JSON files in a `saves/` directory on the server. No database dependency.

**Why not SQLite/Postgres:** The data is a single document per game. There's no relational querying, no concurrent writes to the same game, and no schema that benefits from a database. A JSON file per game is simpler, has zero dependencies, and is easy to inspect and debug.

**File naming:** `saves/{gameId}.json` where `gameId` is a generated unique identifier (not the room code, since room codes are ephemeral and reused).

### Player identity: seat codes + localStorage convenience

Each player gets a **seat code** (short alphanumeric string, e.g., `7X2Q`) when a game is saved. This is their ticket back to their seat.

- **localStorage** stores seat codes as a convenience so players don't have to remember them
- **Seat code entry** is the fallback when localStorage isn't available (different device, cleared storage)
- **sessionStorage** continues to control active in-game identity (unchanged)

### Relationship to the existing localStorage reconnection backlog item

The BACKLOG contains "Reconnection: Survive Tab Closure via localStorage Fallback" which covers reconnecting to an *active* room during the 5-minute grace period. That feature and this one both write to localStorage but serve different purposes:

| | Tab Closure Reconnection (BACKLOG) | Save & Resume (this plan) |
|---|---|---|
| **Timeframe** | Seconds to minutes (within grace period) | Hours to days |
| **Server state** | Room still exists in memory | Room is gone; game is on disk |
| **What's stored** | `sessionToken` + `roomCode` | `{ gameId → { color → seatCode } }` |
| **localStorage key** | `sessionToken`, `roomCode` | `savedGames` |

These use different localStorage keys and don't conflict. The tab closure reconnection feature can be implemented independently (before, after, or alongside this plan). If it lands first, save/resume benefits from it — players get seamless reconnection for short disconnects AND persistent save/resume for long ones.

## What Gets Saved

The save file contains the **minimum state needed to reconstruct the game**. The hex grid is deterministic and regenerated by clients, so it's excluded.

```javascript
{
  gameId: "unique-id",
  savedAt: "2026-03-11T...",
  gameName: "Friday night game",        // Optional, player-provided

  // Seat codes for resuming
  seats: {
    "red": { name: "Alice", seatCode: "7X2Q" },
    "blue": { name: "Bob", seatCode: "M3KP" },
    // ...
  },

  // Game state (subset of gameState)
  state: {
    players: [/* full player objects with demandCards, cash, loads, etc. */],
    currentPlayerIndex: 0,
    turn: 5,
    phase: "build",
    buildingPhaseCount: 0,
    initialBuildingRounds: 2,
    tracks: [],
    ferryOwnership: {},
    demandCardDeck: [],
    activeEvents: [],
    derailedPlayers: {},
    destroyedRiverTracks: [],
    halfSpeedActive: {},
    trackageRightsPaidThisTurn: {},
    trackageRightsLog: [],
    gameLog: []
  }
}
```

**Not saved** (regenerated or reset on resume):
- Hex grid (`mileposts`, `mileposts_by_id`, `cityToMilepost`, `ferryConnections`) — deterministic
- Client UI state (`selectedMilepost`, `pathPreview`, zoom, etc.) — per-session
- `buildHistory`, `operateHistory` — undo stacks reset on resume (mid-turn state is not preserved; see Phase 1 notes)
- `buildingThisTurn`, `majorCitiesThisTurn`, `movement` — turn-scoped state, reset on resume

## Phased Approach

### Phase 0: Decouple Game from Room (~1 day)

**What:** Introduce a `gameId` that identifies a game independently of the room it's hosted in. Currently, game state lives inside the room object and has no identity beyond the room code.

**Changes:**

- Add `gameId` field to the room object (generated at game start via `randomUUID()`)
- `createGameState()` returns a state that includes `gameId`
- Room codes remain ephemeral (used for joining); `gameId` is durable (used for save/resume)

**Risk: Very low.** Additive change — a new field that nothing reads yet.

---

### Phase 1: Server-Side Save/Load (~2-3 days)

**What:** Server can serialize a game to disk and reconstruct it from a save file.

**Save logic:**

1. When save is triggered, generate a seat code for each non-abandoned player
2. Extract the saveable subset of `gameState` (listed above)
3. Write to `saves/{gameId}.json`
4. Return seat codes to each player (each player only sees their own)

**When to save:**

- **Explicit save:** Any player can trigger a save (emits `saveGame` socket event). Save happens at the *end of the current turn* to avoid capturing mid-turn state (partial builds, uncommitted moves). If a save is requested mid-turn, the server queues it and executes after the next `endTurn`.
- **Auto-save (stretch goal):** Server auto-saves at the end of each turn. Overwrites the same file. Players don't need to think about it.

**Load logic:**

1. Read `saves/{gameId}.json`
2. Validate the file structure
3. Create a new room with the loaded game state
4. Set `gameStarted: true` on the room
5. Players join via seat codes (see Phase 2)
6. Once all (non-abandoned) players are seated, the game resumes

**Mid-turn edge case:** Save always happens at a turn boundary. The current player's in-progress builds or moves are *not* saved. This avoids the complexity of serializing undo stacks and partial path state. The save notification should make this clear: "Game will be saved at end of current turn."

**Filesystem considerations:**

- Create `saves/` directory on server start if it doesn't exist
- Add `saves/` to `.gitignore`
- Save files are plain JSON, human-readable for debugging
- No cleanup policy in Phase 1 (manual deletion); auto-expiry is a stretch goal

**Risk: Low.** New endpoints and file I/O. No changes to existing game logic or room management.

---

### Phase 2: Resume Lobby Flow (~2-3 days)

**What:** New lobby UI for resuming saved games, with seat code authentication.

**Server endpoints:**

- `listSavedGames` — returns list of saved games (gameId, gameName, savedAt, player names/colors, player count). No seat codes exposed in the list.
- `resumeGame({ gameId, seatCode })` — validates seat code, loads game if not already loaded, assigns the player to their seat, returns session token + room code

**Client lobby changes:**

The lobby gets a third option alongside "Create Room" and the room list:

```
[Your Saved Games]         ← new section, only visible if localStorage has saved games
  Friday night game (3 players, saved Mar 10)  [Resume]
  Test game (2 players, saved Mar 8)           [Resume]

[Create Room]
[Available Rooms]
  ABCD - Alice's room (2/4 players)  [Join]
```

**Resume flow:**

1. Player clicks "Resume" on a saved game
2. Client checks localStorage for seat codes for that gameId
3. **Single seat found:** Auto-fill seat code, connect immediately
4. **Multiple seats found (testing scenario):** Show seat picker:
   > "Resume as **red** (Alice) or **blue** (Bob)?"
   > [Resume as red] [Resume as blue] [Back to lobby]
5. **No seats in localStorage:** Show manual entry:
   > "Enter your seat code: [____]" [Resume] [Back to lobby]
6. Server validates seat code, loads game into a new room, returns room code
7. Client enters game view via `initializeGameView()` (same path as reconnection)

**Waiting for other players:**

When a player resumes, they enter a waiting room (similar to the pre-game lobby) showing which seats are filled and which are still empty. The game resumes when all non-abandoned seats are claimed. The host (first player to resume, or original host) can optionally start with empty seats marked as abandoned.

**localStorage structure:**

```javascript
// Key: "savedGames"
// Value:
{
  "game-id-1": {
    gameName: "Friday night game",
    savedAt: "2026-03-11T...",
    seats: {
      "red": "7X2Q",
      "blue": "M3KP"
    }
  },
  "game-id-2": { ... }
}
```

All seat codes for a game are stored (not just the player's own), specifically to support the multi-tab testing scenario where one browser controls multiple players. For a normal player who played one seat, only one entry exists under `seats`.

**Risk: Low-medium.** New lobby UI and server logic, but doesn't modify existing create/join/play flows.

---

### Phase 3: localStorage Convenience Layer (~1 day)

**What:** Client automatically saves and retrieves seat codes from localStorage.

**When seat codes are saved to localStorage:**

- After a successful `saveGame` response, the server sends each connected player their seat code
- The client stores it in the `savedGames` localStorage object under the game's `gameId`

**When seat codes are read from localStorage:**

- On lobby load, check `savedGames` for any entries
- If entries exist, show the "Your Saved Games" section
- On resume, auto-fill from localStorage before prompting for manual entry

**Cleanup:**

- After successfully resuming and completing a game (win condition met), remove that game's entry from localStorage
- After a failed resume (game not found on server — save file deleted), remove the stale entry and show a message: "This saved game is no longer available"
- No expiry timer — entries are small and harmless

**Risk: Very low.** Client-only convenience logic.

## Dependencies

### Solo mode

**No blocking dependencies in either direction.** Save/resume and solo mode touch different parts of the codebase:

| | Save/Resume | Solo Mode |
|---|---|---|
| Server: room lifecycle | Adds save/load endpoints, gameId | Adds AI turn loop, AI player slots |
| Server: game logic | No changes | Adds AI decision engine |
| Client: lobby UI | Adds resume section | Adds solo mode button + AI setup |
| Client: game logic | No changes | No changes |
| Shared module extraction | Not needed | Required (Phase 0 of solo plan) |

The only minor overlap is the lobby UI — both add new elements to the lobby screen. If both are in progress simultaneously, coordinate the HTML/CSS to avoid merge conflicts, but there's no logical dependency.

### Tab closure reconnection (BACKLOG item)

**Independent but complementary.** The BACKLOG's localStorage reconnection feature handles short-lived disconnects (tab close during active game, reconnect within 5-minute grace period). Save/resume handles long-lived persistence (game saved to disk, resume hours/days later). They use different localStorage keys and different server paths. Either can be built first.

## How the Product Can Break — Risk Analysis

### Break Case 1: Save file schema drift

**How it happens:** The game state structure changes (new field added, field renamed) but old save files on disk still have the old structure. Loading an old save produces a game state missing required fields.

**Mitigation:** Include a `schemaVersion` in the save file. On load, validate required fields and apply migrations if needed. For v1, a simple "required fields present" check is sufficient — reject saves that don't pass validation rather than attempting complex migrations.

### Break Case 2: Demand card deck state becomes invalid

**How it happens:** The deck is an ordered array of remaining cards. If the card definitions change (rebalanced payouts, added/removed cards) between save and resume, the saved deck references cards that no longer exist or have different properties.

**Mitigation:** Save the full card objects in the deck, not just IDs. The resumed game uses the cards as they were when saved, even if the live card definitions have since changed.

### Break Case 3: Seat code collision

**How it happens:** Two different games generate the same seat code for different players. A player enters a seat code that's valid for the wrong game.

**Mitigation:** Seat codes are scoped to a `gameId`. The resume flow requires both `gameId` and `seatCode`, so collisions across games are impossible. Within a single game (max 6 players), 4-character alphanumeric codes have 1.6M+ combinations — collision probability is negligible.

### Break Case 4: Race condition on save during turn end

**How it happens:** A save is queued to execute at turn end. Two things happen near-simultaneously: the turn ends (triggering save) and a player disconnects (triggering grace timer logic). The save captures a partially-updated state.

**Mitigation:** Save executes synchronously within `serverEndTurn()`, after all turn-end mutations are complete but before broadcasting the new state. The save is a snapshot of a consistent state.

### Break Case 5: Multiple browsers resume the same seat

**How it happens:** A player's seat code is shared (screenshot, written down). Two browsers both try to resume as the same player.

**Mitigation:** First one in wins. Once a seat is claimed in the resume waiting room, the seat code is marked as used. The second attempt gets "This seat is already taken."

### Break Case 6: Stale localStorage entries accumulate

**How it happens:** Player saves many games over weeks. localStorage fills with entries for games that no longer exist on the server.

**Mitigation:** When the lobby loads, the client can send its known gameIds to the server to check which are still valid. Remove stale entries. This is a lightweight check (list of IDs, not full game loads).

## Testing Plan

### Automated Tests

```javascript
// Save/load round-trip
test('saved game state matches loaded game state', () => {
  const original = createGameState(playerList);
  // Simulate several turns of play...
  const saveData = serializeForSave(original);
  const loaded = deserializeFromSave(saveData);
  // Compare all saved fields
  assert.deepStrictEqual(loaded.players, original.players);
  assert.deepStrictEqual(loaded.tracks, original.tracks);
  // ...
});

// Seat code generation
test('seat codes are unique within a game', () => {
  const codes = generateSeatCodes(6); // max players
  const unique = new Set(Object.values(codes));
  assert.strictEqual(unique.size, 6);
});

// Seat code validation
test('valid seat code returns correct player', () => {
  const saveData = createSaveWithSeats({ red: "7X2Q", blue: "M3KP" });
  assert.strictEqual(validateSeatCode(saveData, "7X2Q"), "red");
  assert.strictEqual(validateSeatCode(saveData, "XXXX"), null);
});

// Schema validation
test('rejects save file missing required fields', () => {
  const bad = { gameId: "test", state: { players: [] } }; // missing fields
  assert.throws(() => validateSaveFile(bad));
});

// Mid-turn save queueing
test('save executes after turn end, not mid-turn', () => {
  // Start a turn, queue a save, verify save hasn't happened yet
  // End the turn, verify save has now happened
});
```
